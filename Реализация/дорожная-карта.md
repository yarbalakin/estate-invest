## Волна 0 — Реструктуризация (эта неделя)

> Цель: разделить перегруженный отдел на два с чёткими зонами ответственности
> Подробности: см. `структура-отделов.md`

### 0.1 Разделение на Отдел реализации + Отдел продаж
**Проблема**: на одних людей навалены две разных работы — оформление/улучшение (юридическая + стройка) и продажа (коммерческая). Конфликт приоритетов, перегруз.
**Решение**:
- **Отдел реализации** (Иван): выиграли торги → конкурсный → регистрация ДКП → снятие обременений → право собственности → улучшение (межевание, ремонт, ВРИ, раздел) → объект готов
- **Отдел продаж** (Наталья): объект готов → объявления → показы → переговоры → сделка → статистика
**Точка передачи**: чёткий чек-лист (право зарегистрировано, обременения сняты, фото, цена, документы)
**Усилия**: 1-2 дня (обсуждение + формализация)
**Ответственный**: Ярослав + Иван + Наталья

### 0.2 Решение по улучшениям объектов
**Проблема**: улучшение — плотная работа (курирование строителей, снабжение, межевание), делается на энтузиазме Ивана
**Варианты**:
- A: Простые улучшения → риэлтор/помощник, сложные → Иван
- B: Нанять прораба/куратора (40-60К/мес)
- C: Гибрид (рекомендуется): пилот с существующей командой, потом прораб
**Критерий "простой объект"**: ремонт < 200К, не требует юридических действий
**Усилия**: решение за 1 день, пилот за 2 недели
**Ответственный**: Иван (оценка), Ярослав (решение)

### 0.3 Пересмотр математики на реализацию
**Проблема**: если улучшение — полноценная работа, а не "бесплатный энтузиазм", нужно заложить это в финмодель
**Действия**:
- Посчитать среднее время Ивана на улучшение 1 объекта (часы/неделю)
- Посчитать стоимость этого времени
- Заложить в финмодель как расход (даже если платит себе Иван)
- Определить: при каком количестве объектов окупается наём куратора
**Усилия**: 2-3 часа (расчёт)
**Ответственный**: Иван (данные) + Ярослав (расчёт)

## Волна 1 — Быстрые победы (1-2 недели)

> Цель: убрать самые болезненные ручные операции без программирования

### 1.1 Чек-лист приёмки объекта
**Проблема**: при создании нового объекта нужно обновить 6 систем — всё в голове у Ивана
**Решение**: создать шаблон чек-листа (Google Docs или Notion) со всеми шагами БП-1
**Эффект**: ничего не забывается, можно делегировать Андрею или Жанне
**Усилия**: 1 час
**Ответственный**: Ярослав (создать шаблон) + Иван (проверить)

### 1.2 Чек-лист передачи "Аналитика → Реализация"
**Проблема**: передача между отделами "на словах", теряются документы и данные
**Решение**: фиксированный список того, что Аналитика ОБЯЗАНА передать перед стартом работ
**Содержание**: адрес, кадастровый номер, финмодель, правоустанавливающие, план работ, бюджет, сроки
**Усилия**: 30 минут
**Ответственный**: Ярослав

### 1.3 Чек-лист передачи "Реализация → Продажи"
**Проблема**: Наталья получает объекты без стандартного набора материалов
**Решение**: фиксированный список: фото (min 10 шт, формат), описание объекта, цена, документы, контакт Ивана
**Усилия**: 30 минут
**Ответственный**: Иван + Наталья

### 1.4 Автонапоминания о дедлайнах торгов
**Проблема**: Жанна вручную считает дату задатка (3 рабочих дня до окончания подачи заявок)
**Решение**: n8n workflow — берёт даты из таблицы торгов → рассчитывает дедлайны → шлёт напоминания в Telegram
**Эффект**: нулевой риск пропустить дедлайн
**Усилия**: 2-3 часа (n8n workflow)
**Ответственный**: Ярослав (настроить n8n)

### 1.5 Шаблон объявления для Авито
**Проблема**: каждое объявление пишется с нуля, нет стандарта
**Решение**: 3-4 шаблона по типам (земля, квартира, коммерческая, дом) — заполняешь поля, получаешь текст
**Эффект**: создание объявления 10 мин вместо 40 мин
**Усилия**: 2 часа
**Ответственный**: Наталья (контент) + Ярослав (шаблон)

### 1.6 Система учёта коммунальных услуг
**Проблема**: КУ считают на листочке, Excel как таблица без формул. Наталья вручную считает долю каждого арендатора, напоминает, отправляет квитанцию с QR. При 52 объектах — серьёзная рутина.
**Логика расчёта** (на примере Нытвы):
- Приходит общий счёт на всё здание (подвал + этажи)
- Общая сумма / общая площадь = стоимость за м²
- Сданные помещения: арендатор платит за свои м²
- Коридоры и свободные помещения: оплачивает компания
**Данные по Нытве**: 1-3 этаж = 580.9 м², подвал = 289.8 м², итого 870.7 м²
**Решение**: HTML-калькулятор → ввёл общую сумму → автоматический расчёт долей → готовые сообщения для арендаторов с суммами → отправка в WhatsApp/Telegram
**Эффект**: расчёт за 30 секунд вместо 20 минут, нулевой риск ошибок в арифметике
**Усилия**: 1-2 дня
**Ответственный**: Ярослав (инструмент) + Наталья (тестирование)
**Пилот**: объект Нытва (12 арендаторов, 2 этажа)
**Масштабирование**: после пилота — добавить остальные объекты (Солдатова 32, и др.)
**Будущее**: интеграция с Telegram-ботом для автоматических напоминаний и приёма чеков об оплате

## Волна 2 — Автоматизация рутины (2-4 недели)

> Цель: автоматизировать повторяющиеся действия через n8n и скрипты

### 2.1 Автообновление статусов (синхронизация систем)
**Проблема**: при смене стадии объекта нужно обновить Монитор + Notion + ЛК + Битрикс — 4 системы вручную
**Решение**: n8n workflow — изменение статуса в одном месте (Монитор) → автоматическое обновление остальных
**Вариант реализации**:
- Триггер: Google Sheets webhook (изменение столбца "статус" в Мониторе)
- Действия: обновить Notion (API), обновить Битрикс (REST API), уведомить Telegram
- ЛК синхронизируется из основной таблицы учёта
**Эффект**: экономия 15-20 мин на каждую смену статуса, нет рассинхрона
**Усилия**: 1-2 дня
**Ответственный**: Ярослав

### 2.2 Календарь торгов с автоматическим расчётом дат
**Проблема**: Жанна вручную мониторит Тбанкрот и рассчитывает даты
**Решение**: расширение torgi_monitor.py — добавить расчёт:
- Дата аукциона / окончания этапа
- Дата окончания подачи заявок
- Дата задатка (минус 3 рабочих дня)
- Сохранение в отдельный Google Sheet "Календарь торгов"
**Эффект**: Жанна видит готовый календарь, не нужно считать вручную
**Усилия**: 1 день
**Ответственный**: Ярослав

### 2.3 Массовое создание объявлений из данных
**Проблема**: 118 объектов, но только 46 на Авито (39%)
**Решение**: скрипт, который берёт данные из Монитора (тип, адрес, площадь, цена, фото) → генерирует текст по шаблону → готовый черновик для Авито
**Эффект**: подготовка объявления автоматически, Наталья только проверяет и публикует
**Усилия**: 2-3 дня
**Ответственный**: Ярослав (скрипт) + Наталья (проверка)

### 2.4 Реестр подрядчиков
**Проблема**: контакты подрядчиков только в телефоне Ивана
**Решение**: Google Sheet с колонками: имя, специализация, регион, контакт, рейтинг, комментарий
**Эффект**: любой в команде может найти подрядчика, Иван не bottleneck
**Усилия**: 2-3 часа (Иван заполняет)
**Ответственный**: Иван

### 2.5 Уведомления инвесторам о смене стадии
**Проблема**: инвесторы не знают, что происходит с объектом, пока не спросят
**Решение**: при смене статуса в Мониторе → автоуведомление через Salebot инвесторам этого ИП
**Пример**: "ИП №95 перешёл в стадию 'Подготовка к выплате'. Ожидайте расчёт доходности."
**Эффект**: снижение нагрузки на отдел поддержки, инвесторы лояльнее
**Усилия**: 1-2 дня (n8n + Salebot API)
**Ответственный**: Ярослав

## Волна 3 — Единая система (1-2 месяца)

> Цель: от разрозненных таблиц к единой панели управления отделом

### 3.1 Канбан-доска объектов
**Проблема**: нет визуального представления "воронки объектов"
**Решение**: визуальная доска (Битрикс24 задачи или custom HTML):
```
Торги → Сбор средств → Выкуп → Ремонт/Раздел → Продажа → Закрыт
```
- Каждый объект — карточка с ключевой инфой
- Перетаскивание между стадиями = автообновление всех систем (через 2.1)
- Фильтры: по ответственному, типу, региону
**Варианты реализации**:
- A: Битрикс24 (уже есть, настроить новую воронку)
- B: Notion Board (простой канбан)
- C: Custom HTML (как реализация-сводка.html, но с drag-and-drop)
**Рекомендация**: вариант A (Битрикс) — уже есть воронка "Эстейт продажи", расширить до полного цикла
**Усилия**: 1-2 недели
**Ответственный**: Ярослав (настройка) + Иван (валидация стадий)

### 3.2 CRM покупателей
**Проблема**: покупатели объектов нигде не фиксируются — звонки, показы, переговоры теряются
**Решение**: использовать воронку "Эстейт продажи" (ID: 8) в Битрикс24 полноценно:
- Каждый звонок = контакт + сделка
- Стадии: Заявка → Показ → Переговоры → Задаток → Сделка / Отказ
- Наталья и Наталья ведут все контакты в Битрикс
**Эффект**: видно, сколько покупателей на каждом объекте, на каком этапе
**Усилия**: 1 неделя (настройка + обучение)
**Ответственный**: Наталья + Наталья

### 3.3 Цифровой трекер ремонта
**Проблема**: нет контроля этапов ремонта, бюджета факт/план
**Решение**: для каждого объекта в ремонте — карточка с:
- Список работ (этап / срок / бюджет план / бюджет факт / статус)
- Фотоотчёт по этапам
- Автоалерт если работы затягиваются > 20% от плана
**Вариант**: Google Sheet с формулами + n8n алерты
**Усилия**: 1 неделя
**Ответственный**: Иван (данные) + Ярослав (настройка)

### 3.4 Доступ к облаку Mail.ru через WebDAV
**Проблема**: документы (выписки ЕГРН, договоры, сметы) хранятся в облаке Mail.ru, доступ только вручную
**Решение**: подключить WebDAV API (`https://webdav.cloud.mail.ru/`) — AI-ассистент может сам находить и читать документы
**Что даёт**: "Открой выписку ЕГРН по Нытве и найди общую площадь" — без ручного поиска
**Что нужно**: создать пароль приложения в настройках безопасности Mail.ru
**Усилия**: 30 минут (настройка)
**Ответственный**: Ярослав

### 3.5 Единое хранилище документов
**Проблема**: документы в облаке Mail.ru, Google Drive, локально — невозможно найти
**Решение**: единая структура в Google Drive:
```
Объекты/
  ИП №115/
    Торги/ (протоколы, заявки)
    Правоустанавливающие/ (выписки, договоры)
    Ремонт/ (сметы, фото до/после)
    Продажа/ (объявления, договоры КП)
```
- Ссылка на папку — в карточке объекта (Монитор/Notion)
**Усилия**: 2-3 дня (создать структуру + перенести текущие)
**Ответственный**: Наталья + Жанна

### 3.5 Аналитика продаж
**Проблема**: нет данных об эффективности продаж — какие объявления работают, какие нет
**Решение**: дашборд (HTML или Streamlit):
- Объектов по стадиям (воронка)
- Среднее время на каждой стадии
- Конверсия показ → сделка
- ТОП-5 зависших объектов (дольше всех на продаже)
- ТОП-5 быстрых продаж (для понимания паттерна)
**Источник данных**: Монитор объектов + Битрикс24
**Усилия**: 1-2 недели
**Ответственный**: Ярослав

## Волна 4 — AI-слой (2-4 месяца)

> Цель: AI берёт на себя рутину, люди принимают решения. Соответствует Фазе 2 из AI-трансформации.md.

### 4.1 AI-генерация объявлений
**Проблема**: написание объявлений — рутинная, но творческая работа
**Решение**: из карточки объекта (тип, адрес, площадь, кадастровый номер, фото) → AI пишет:
- Заголовок (3 варианта для A/B тестирования)
- Описание (по шаблону, но адаптированное)
- Рекомендованная цена (на основе аналогов)
**Стек**: Python + OpenRouter/Claude API + данные из Монитора
**Усилия**: 1-2 недели
**Ответственный**: Ярослав

### 4.2 AI-оценщик объектов
**Проблема**: определение цены продажи — субъективно, на основе опыта Ивана
**Решение**: AI-модель на данных 120+ реализованных объектов:
- Вход: тип, регион, площадь, цена покупки, состояние
- Выход: диапазон рыночной цены, рекомендуемая цена, прогноз срока продажи
- Источники аналогов: Авито-парсер, данные компании
**Уже есть**: аналоговая карта (analog_map.html), скоринг (enrich_scoring.py), геокодинг — можно расширить
**Усилия**: 2-4 недели
**Ответственный**: Ярослав

### 4.3 Автоматические отчёты по объектам
**Проблема**: партнёры не видят общую картину без ручного сбора данных
**Решение**: еженедельный AI-дайджест в Telegram:
```
Отдел реализации — отчёт за неделю:
- В ремонте: 5 объектов (ИП №95 — задержка 12 дней)
- На продаже: 52 объекта (3 новых показа)
- Продано за неделю: ИП №100 (30% ROI)
- Зависшие: ИП №35 (>90 дней на продаже)
- Следующая неделя: 2 торга, 3 задатка
```
**Стек**: расширение pulse_estate.py или отдельный n8n workflow
**Усилия**: 1 неделя
**Ответственный**: Ярослав

### 4.4 AI-советник по стратегии продаж
**Проблема**: нет систематического подхода к зависшим объектам
**Решение**: AI анализирует объекты, которые на продаже > 60 дней, и предлагает:
- Снижение цены (на сколько, обоснование)
- Смена площадки (добавить ЦИАН, Домклик)
- Обновление фото/описания
- Целевая аудитория (кому интересен этот объект)
**Усилия**: 2 недели
**Ответственный**: Ярослав

# Критический разбор исследования — Честная обратная связь

> 25.02.2026. Взгляд опытного разработчика на собранные материалы.

---

## ГЛАВНАЯ ПРОБЛЕМА: Исследование описывает три разных продукта одновременно

В файлах смешаны три принципиально разных вещи:

1. **Статичная HTML-страница** (как текущий dashboard.html) — открыл в браузере, посмотрел
2. **SaaS-дашборд уровня Stripe/Linear** — с real-time обновлениями, drill-down, канбаном
3. **BI-платформа** — 5 источников данных, 50 виджетов, 9 секций, алерты

Это НЕ одно и то же. Нельзя "сделать single HTML file" и одновременно "тянуть данные из 5 API с кэшированием каждые 10 минут". Нужно выбрать одно.

## ПРОБЛЕМА 1: Масштаб не соответствует реальности

### Что написано:
- 50 виджетов (25 must-have)
- 9 секций навигации
- 5 источников данных
- Канбан, карта Leaflet, календарь, funnel charts, heatmap, реферальный граф

### Реальность:
- Дашборд смотрят **2 человека** (Ярослав + Иван)
- Это НЕ продукт для 1000 пользователей, это внутренний инструмент
- 25 "must-have" виджетов — это НЕ приоритизация, это wish list
- Для MVP достаточно **одной страницы с 7 KPI + 3 графика + 1 таблица**

### Что делать:
Честный must-have для первой версии — **максимум 10-12 виджетов на одном экране**. Остальное — v2, v3, v4. Или никогда.

## ПРОБЛЕМА 2: "Single HTML file" — это ложь при таком скоупе

### Что написано:
"Стек: чистый CSS, Vanilla JS, без фреймворков, single HTML file"

### Реальность при 50 виджетах:
- HTML: ~3000-5000 строк
- CSS: ~2000-3000 строк
- JS: ~2000-4000 строк (графики, анимации, загрузка данных, фильтры)
- **Итого: один файл на 8000-12000 строк**

Это уже не "простой HTML", это приложение, которое невозможно поддерживать. Один файл на 12K строк — любое изменение = мучение.

### Что делать:
Либо принять, что это будет несколько файлов (index.html + style.css + app.js + data.js), либо радикально сократить скоуп до того, что реально помещается в 1 файл (1 экран, 10-12 виджетов, ~2000 строк).

## ПРОБЛЕМА 3: Данные — самое сложное, а не дизайн

### Что написано:
80% исследования — про CSS, цвета, шрифты, анимации. 20% — про данные.

### Реальность:
80% времени разработки уйдёт на ДАННЫЕ, не на дизайн:
- Google Sheets gviz endpoint для 76K строк — **медленно** (10-30 сек)
- Битрикс24 REST API — пагинация по 50 записей, для 1142 сделок = 23 запроса последовательно
- Мультивалютность — парсинг курсов из строк 1-3 таблицы = хрупкая конструкция
- Кэширование — где? В localStorage? Service Worker? Серверный кэш?
- CORS — Google Sheets gviz endpoint и Битрикс REST могут блокировать browser-side запросы

### Что делать:
Нужен отдельный файл `09-архитектура-данных.md` с честным разбором:
- Какие данные реально можно загрузить из браузера (CORS)
- Нужен ли бэкенд/прокси (спойлер: скорее да)
- Как кэшировать 76K строк
- Стратегия обновления (вручную кнопкой, не polling)

## ПРОБЛЕМА 4: Три разных цветовых палитры + несогласованность

- Файл 01: фон `#0a0a0f`, текст `#e4e4e7`, акцент `#8170ed`
- Файл 03: фон `#0a0a0f`, текст `#f1f5f9`, акцент `#6366f1`
- Файл 03 alt: фон `#0f172a`, акцент `#6366f1` (Midnight Indigo)
- Файл 03 alt2: фон `#030712`, акцент `#38bdf8` (Ocean Deep)

- Файл 07: фон `#0f172a`, текст `#f1f5f9`, акцент `#3b82f6`

### Реальность:
Шесть разных фоновых цветов, четыре разных акцента, три разных основных текста. Это не "выбор", это хаос. При реализации разработчик потратит час только на то, чтобы понять какая палитра правильная.

### Что делать:
**ОДНА палитра. Точка.** Записать в отдельный файл `palette.css` и больше не менять. Рекомендация: взять Tailwind Slate (файл 07) — она проверена миллионами пользователей.

## ПРОБЛЕМА 5: Новогодняя ёлка из эффектов

### Что предлагается использовать ОДНОВРЕМЕННО:
1. Glassmorphism (backdrop-filter blur)
2. Noise/grain overlay
3. Aurora animated background (3 блоба с blur 80px)
4. Glow на hover
5. 3D tilt на hover
6. Gradient borders
7. Многослойные тени (6 уровней)
8. Animated gradient shift
9. Shine-эффект (полоска блика)
10. Каскадный fadeInUp
11. Count-up анимация чисел
12. SVG draw-in для графиков
13. Skeleton shimmer
14. Пульсирующие статус-точки

### Реальность:
Stripe, Linear и Vercel выглядят premium потому что используют **2-3 приёма из этого списка**, не все 14. Исследование само же цитирует: "Linear: less is more". И тут же предлагает "а давайте добавим всё".

`backdrop-filter: blur(80px)` на 3 блобах + `blur(12px)` на каждой из 50 карточек = **GPU будет плакать**. Особенно на MacBook Air.

### Что делать:
Выбрать МАКСИМУМ 4-5 эффектов:
1. Тёмный фон с тонкими radial-gradient orbs (статичные, не анимированные)
2. Карточки с простым background + border (без blur)
3. Hover: translateY(-2px) + shadow
4. fadeInUp при загрузке
5. Анимация чисел

Всё остальное — выкинуть. Серьёзно.

## ПРОБЛЕМА 6: "Must-have" значит всё и ничего

### В контент-карте (05):
25 виджетов помечены "Must-have". Это включает:
- Канбан-доску объектов (это отдельное приложение)
- P&L карточку с модальным окном (это мини-CRM)
- 4 KPI-карточки мультивалютного баланса (4 API-запроса за курсами)
- Funnel chart из Битрикс (23 запроса к API)
- Таблицу с поиском и фильтрами (это DataGrid компонент)

### Реальность:
Канбан на vanilla JS — это ~500-1000 строк кода. Таблица с поиском/сортировкой/фильтрами — ещё ~500. Модальное окно с P&L — ещё ~300. Только эти три виджета = больше кода, чем весь текущий dashboard.html.

### Что делать:
Переопределить "Must-have" как: **то, что можно сделать за 1 день**. Всё остальное — "want-to-have". Честный must-have:
1. 5-7 KPI-карточек (статичные числа, обновление вручную)
2. 1 area chart (динамика капитала)
3. 1 bar chart (объекты по стадиям)
4. 1 donut (распределение по типам)
5. 1 таблица (последние 10-20 объектов)

## ПРОБЛЕМА 8: Дублирование между файлами

### Примеры:
- KPI-карточка CSS описана в файлах 01, 02, 03 — каждый раз немного по-разному
- Палитра описана в 01, 03, 07 — с разными hex-кодами
- Анимация fadeInUp — в 01, 03, 07

### Что делать:
При реализации нужен ОДИН файл-источник истины с финальными решениями, а не 7 файлов с вариациями одного и того же.

## ПРОБЛЕМА 9: Нет ответа на вопрос "откуда берутся данные"

### Что есть:
Spreadsheet ID таблиц, endpoints Битрикс. Звучит хорошо.

### Чего нет:
- Как загрузить 76K строк из Google Sheets в БРАУЗЕР? (gviz endpoint отдаёт CSV, парсить в JS?)
- CORS: браузер → Google Sheets gviz — работает? (спойлер: обычно да, но нестабильно)
- CORS: браузер → Битрикс REST API — работает? (спойлер: обычно нет без настройки)
- Что если Sheets API меняет формат? Что если endpoint падает?
- Аутентификация: дашборд с финансами компании — открытый HTML файл? Любой с ссылкой видит всё?
- Размер данных: 76K строк × ~20 колонок = ~5-15MB CSV. На каждую загрузку страницы?

## ИТОГО: Что реально делать

### Правильный порядок:

**Шаг 1** — Решить архитектуру (30 мин):
- Статический HTML с ручным обновлением данных? (проще, но нужно обновлять руками)
- HTML + JS с загрузкой из Google Sheets? (автоматические данные, но CORS/размер/скорость)
- Python бэкенд (уже есть data_loader.py!) + HTML фронт? (надёжнее, но сложнее деплой)

**Шаг 2** — Сделать MVP (1 день):
- Один экран, 5-7 KPI, 2-3 графика, 1 таблица
- Данные — hardcoded или из одного источника
- ОДНА палитра, ОДНА анимация, НОЛЬ blur-эффектов

**Шаг 3** — Подключить данные (2-3 дня):
- Начать с Google Sheets (основной источник)
- Потом Битрикс (если CORS позволяет)
- Кэш в localStorage

**Шаг 4** — Наводить красоту (1 день):
- Добавить 2-3 эффекта из файла 07
- Проверить на реальном устройстве
- Убрать то, что тормозит или отвлекает

**Шаг 5** — Итерировать (бесконечно):
- Добавлять секции по одной, не все 9 сразу
- Каждую новую секцию — тестировать, получать фидбэк, потом следующую

## Что в исследовании ХОРОШО

Чтобы не было только критики — вот что реально ценно:

1. **Контент-карта (файл 05)** — отличная декомпозиция, привязка к реальным данным. Нужно только урезать скоуп.
2. **Живые примеры (файл 04)** — 27 ссылок, можно открыть и потрогать. Особенно Tabler и ApexCharts Dark.
3. **Антипаттерны (файл 06)** — список ошибок ценнее списка "лучших практик". Запомнить и не повторять.
4. **Анатомия KPI-карточки (файл 01)** — конкретная структура из 5 элементов. Можно брать и делать.
5. **ApexCharts рекомендация** — правильный выбор, покрывает все нужные типы графиков.
6. **`tabular-nums`** — мелочь, но критичная для дашборда. Правильный совет.
7. **Правило 5-7 KPI (Miller's Law)** — важное ограничение. Следовать ему.

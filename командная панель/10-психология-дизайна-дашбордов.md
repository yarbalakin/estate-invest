# Психология и когнитивная наука дизайна дашбордов

> Научная база: 30+ источников, eye-tracking исследования, когнитивная психология

---

## 1. Преаттентивные атрибуты (обработка за <500мс)

Мозг обрабатывает определённые свойства за 200-500мс **без сознательного усилия** (Colin Ware, "Visual Thinking for Design").

| Атрибут | На дашборде |
|---------|-------------|
| **Цвет** | Красный среди серых = мгновенное внимание к проблеме |
| **Размер** | Hero KPI 48-72px, второстепенные 24-32px |
| **Ориентация** | Стрелки трендов вверх/вниз |
| **Форма** | Круглые = норма, треугольные = предупреждение |

**Pop-out эффект**: когда один элемент отличается по ОДНОМУ атрибуту от остальных — мгновенно привлекает внимание. Все метрики серые, проблемные — красные.

**Правило**: максимум 2-3 преаттентивных канала одновременно. Больше = перегрузка.

## 2. Гештальт-принципы

### 6 принципов и их применение

1. **Близость**: связанные KPI группируй в одну карточку. Белое пространство между группами > внутри (2:1)
2. **Сходство**: все метрики дохода — синие, расходы — оранжевые. По всему дашборду
3. **Замкнутость**: достаточно двух осей. Не нужны полные рамки — лёгкая тень
4. **Непрерывность**: элементы по строгой сетке. Трендовые линии — мозг "продолжает" тренд
5. **Фигура-фон**: карточки = фигура (приподняты тенью), фон = нейтральный
6. **Общая область**: карточки (cards) — самый эффективный способ группировки

## 3. Когнитивная нагрузка

### Закон Миллера
Рабочая память: **7±2 элемента**. Реальный предел: **~4 элемента** (Cowan, 2001).

### Мантра Шнейдермана (1996)
**"Overview first, zoom and filter, then details-on-demand"**

1. **Обзор**: 5-7 ключевых метрик, одним взглядом
2. **Исследование**: клик раскрывает тренд, разбивку
3. **Детали**: drill-down в конкретные записи

### Data-Ink Ratio (Tufte)
Удаляй всё, что не несёт информации. Но не до абсурда — лёгкие разделители и иконки помогают навигации.

### Спарклайны (Tufte)
Маленькие inline-графики размером со слово. "Datawords" — увеличивают плотность информации без когнитивной нагрузки.

**Правило**: 5-9 визуализаций на экран, сгруппированных по 3-4 чанка.

## 4. Психология цвета

### Три типа палитр (ColorBrewer)

| Тип | Когда | Пример |
|-----|-------|--------|
| **Последовательная** | Низкое → высокое | Светло-синий → тёмно-синий |
| **Расходящаяся** | Отклонения от центра | Красный ← серый → синий |
| **Качественная** | Категории без порядка | Синий, оранжевый, зелёный |

### Дальтонизм (КРИТИЧНО)
- **8% мужчин** — красно-зелёная слепота
- Красный+зелёный как единственные индикаторы = **исключить**
- **Безопасные пары**: синий+оранжевый, синий+красный, синий+коричневый

### Colorblind-safe палитры
- **Okabe-Ito** (8 цветов): `#E69F00, #56B4E9, #009E73, #F0E442, #0072B2, #D55E00, #CC79A7, #000000`
- **Tableau Color Blind 10**
- Инструменты: [ColorBrewer 2.0](https://colorbrewer2.org/), Color Oracle

### Правила
1. **Не кодируй ТОЛЬКО цветом** — добавляй иконку/форму/текст
2. **Максимум 5-7 цветов** в одной визуализации
3. **Серый — основа**: контекст, второстепенные элементы
4. **Проверяй через симулятор**: Chrome DevTools → Rendering → Emulate vision

## 5. Типографика и числа

### Tabular vs Proportional
**Табличные цифры** — одинаковая ширина для всех цифр. Критично для колонок, счётчиков, сравнений.

```css
.dashboard-number {
  font-variant-numeric: tabular-nums lining-nums;
}
```

### Рекомендуемые шрифты
- **Inter** — создан для экранов, tabular через CSS
- **Roboto** — стандарт Material Design
- **SF Pro** — Apple экосистема

### Форматирование чисел
| Контекст | Формат | Пример |
|----------|--------|--------|
| KPI-карточка | Сокращённый | **1.2M**, **34.5K** |
| Таблица | С разделителями | **1,234,567** |
| Tooltip | Полный | **1,234,567 просмотров** |

### Иерархия размеров
1. **Hero-число**: 48-72px, bold
2. **Вторичные метрики**: 24-32px, semibold
3. **Подписи**: 12-14px, regular, приглушённый
4. **Таблицы**: 13px, line-height 1.4

## 6. Eye-tracking: куда смотрит глаз

### F-паттерн (Nielsen, 2006)
1. Горизонтальное сканирование верхней части
2. Второе горизонтальное — ниже, короче
3. Вертикальное — левая сторона

### Z-паттерн
Для визуально разреженных дашбордов: верхний левый → правый → диагональ → нижний левый → правый.

### Где размещать на дашборде

```
┌─────────────────────┬─────────────────────┐
│ САМОЕ ВАЖНОЕ        │ ВТОРОЕ ПО ВАЖНОСТИ  │
│ (основной KPI)      │ (тренд, сравнение)  │
├─────────────────────┼─────────────────────┤
│ КОНТЕКСТ            │ ДЕТАЛИ              │
│ (графики трендов)   │ (таблицы)           │
└─────────────────────┴─────────────────────┘
```

**Верхний левый** = первая точка фиксации глаза = самая важная метрика.

## 7. Когнитивные искажения и дашборды

### Три главных

| Искажение | Как дашборд усиливает | Как снизить |
|-----------|----------------------|-------------|
| **Якорь** | Hero-число задаёт фрейм | Показывай контекст: среднее, медиану, диапазон |
| **Доступность** | Только последние 7 дней → забывают долгосрочное | Несколько таймфреймов (день/неделя/месяц/год) |
| **Подтверждение** | Фильтруют чтобы увидеть желаемое | Проактивно показывай контринтуитивные данные |

### Противоядие
1. Бенчмарк рядом с каждым числом
2. Множественные временные горизонты
3. Автоматические аномалии (система подсвечивает)
4. Нейтральный язык: "Изменение: -12%" вместо "ПАДЕНИЕ!"
5. Tooltip с методологией

## 8. Dark Mode vs Light Mode

| Сценарий | Режим | Почему |
|----------|-------|--------|
| Мониторинг / NOC | Dark | Комфорт для длинных смен |
| Тёмное помещение | Dark | Меньше бликов |
| TV-дашборд в офисе | Dark | Лучше выглядит |
| Яркий офис | Light | Лучше читаемость |
| Много текста/таблиц | Light | Текст читается быстрее |
| Печать | Light | Экономия тонера |

**Правила**:
- По умолчанию: `prefers-color-scheme` (системные настройки)
- Dark: фон `#1a1a2e`, текст `#e0e0e0`
- В dark mode снижай насыщенность ярких цветов на 10-20%
- Контраст >= 4.5:1 (WCAG)
- Не используй чистый чёрный #000 и белый #FFF

## 9. Анимация и переходы

### Когда помогает
- **Change blindness**: без анимации пользователь не замечает обновление числа
- **Переходы**: фильтр → данные плавно перестраиваются = понятно
- **Onboarding**: последовательное появление элементов

### Когда вредит
- Постоянное мигание = отвлекает
- Длинные анимации >600мс = раздражают
- Анимация каждого real-time обновления = мельтешение

### Параметры
- Длительность: **200-400мс** (оптимум 300мс)
- Easing: ease-out для появления
- Real-time: анимировать только значимые изменения (>5%)
- Счётчики: countUp 400-600мс

### Микро-взаимодействия
1. Hover на карточке: тень +2-4px за 150мс
2. Обновление числа: fade out 100мс → fade in 200мс
3. Алерт: slide-in 300мс + pulse 1 раз
4. Loading: skeleton shimmer (не спиннер)

**Уважай `prefers-reduced-motion`** — отключай анимации.

## 10. Glanceability: тест 5 секунд

### Определение (Stephen Few)
"Визуальное отображение важнейшей информации на одном экране так, чтобы её можно было отслеживать одним взглядом."

### 5-Second Test
1. Покажи дашборд на 5 секунд
2. Скрой
3. Спроси: "Какой статус? Какие метрики? Есть проблемы?"
4. Если не может ответить — **упрощай**

### Архитектура glanceable дашборда

```
ВЕРХ:   Статусная полоса (3-5 индикаторов)
        → "Всё ли в порядке?"

ЦЕНТР:  2-3 Hero-числа с дельтами и спарклайнами
        → "Какие ключевые показатели?"

НИЗ:    Графики и таблицы
        → "Почему и что делать?"
```
